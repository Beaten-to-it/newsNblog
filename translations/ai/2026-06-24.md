## [ChatGPT 광고를 정의하는 것은 무엇인가? OpenAI가 칸 라이언스에서 밝힌 '초고의도 사용자'](https://www.marketingdive.com/news/how-openai-positioning-chatgpt-ads-cannes-lions-debut/823476/)

OpenAI는 이번 주 광고 업계 최대 행사인 칸 라이언스(Cannes Lions)에 공식 데뷔하며, 자사 광고 플랫폼을 기존 디지털 마케팅과 차별화된 제품으로 소개했다. OpenAI의 글로벌 광고 솔루션 총괄 데이브 두건(Dave Dugan)은 "완전히 새로운 광고 제품과 완전히 새로운 경험"을 공개했다고 밝혔다.

### 주목경제에서 지능경제로

두건은 ChatGPT 사용자가 콘텐츠를 스크롤하는 것이 아니라 정보와 해결책을 찾기 위해 들어온다는 점을 강조했다. 이는 전통적인 소셜 미디어와 ChatGPT를 근본적으로 다른 광고 지면으로 만든다는 논리다. OpenAI의 최고수익책임자(CRO) 데니스 드레서(Denise Dresser)는 이를 보다 직접적으로 표현했다. "우리는 주목경제(attention economy)에서 지능경제(intelligence economy)로 이동하고 있으며, 소비자가 진정으로 원하는 것은 유용함입니다."

### '퍼널 중간'에 놓인 광고 지면

ChatGPT 광고의 핵심 포지셔닝은 '마케팅 퍼널의 중간 단계', 즉 소비자가 제품과 서비스를 리서치하고 비교하는 결정 레이어다. 두건은 "사람들이 해결해야 할 일을 들고 들어옵니다. 매우 의도적입니다. 그들은 자신이 풀려고 하는 문제에 대한 정보를 우리에게 줍니다"라고 설명했다. OpenAI에 따르면 ChatGPT 질문 가운데 약 20%가 직접적인 상업적 의도를 보인다. 나머지 사용자 행동에서도 상위 퍼널 참여에서 마케터가 활용할 수 있는 예측 기회가 존재한다는 설명도 덧붙였다.

### 플랫폼 초기 성과 지표

올해 2월 출시 이후 ChatGPT 광고는 일본, 한국을 포함한 7개 시장으로 확대됐으며, 라틴 아메리카 확장도 계획 중이다. 광고 네트워크 크리테오(Criteo)에 따르면 현재 2,000개 이상의 브랜드가 이 플랫폼에서 광고를 집행하고 있다.

주목할 만한 성과 지표로는 출시 이후 광고 닫기율(cross-out rate)이 50% 감소했다는 점과, 다른 채널 대비 트래픽 품질이 높다는 일화적 보고가 있다. OpenAI는 자체 관리 도구와 원래의 CPM(노출 1천 회당 비용) 모델에 더해 CPC(클릭당 비용) 입찰 방식도 도입했다.

### 신뢰 원칙과 측정

OpenAI는 광고 접근 방식을 이끄는 네 가지 원칙으로 ① 스폰서 콘텐츠와 일반 응답의 분리, ② 사용자 프라이버시 보호, ③ 사용자 통제권 유지, ④ 장기적 가치 제공을 제시했다. 두건은 "대화 정보를 광고주와 절대 공유하지 않는다"고 명확히 선을 그었다. OpenAI는 브라우저 기반 추적을 넘어 측정 역량을 강화하기 위해 라이브램프(LiveRamp)와도 파트너십을 맺었다.

### 크리에이티브 제작의 변화

OpenAI의 크리에이티브 전문가 채드 넬슨(Chad Nelson)은 브랜드들이 이제 수천 개가 아니라 "수만 개의 에셋"을 제작한다고 밝혔다. 이는 코딩 전문 지식 없이도 사용할 수 있는 새로운 제작 도구와 워크플로를 필요로 한다는 뜻이다.

### 왜 지금 이 발표가 중요한가

ChatGPT가 광고 플랫폼이 되면 AI 어시스턴트의 수익 모델, 사용자 신뢰, 답변 독립성 논쟁이 한꺼번에 커진다. OpenAI는 광고를 무료·하위 티어 구독자를 위한 비용 보조 수단으로 보고 있으며, 주간 활성 사용자 9억 명 이상을 기반으로 상업적 의도가 있는 리서치 활동에서의 수익화 가능성을 크게 본다. 동시에 사용자가 AI 답변을 조언으로 받아들이는 맥락에서 광고와 응답을 어떻게 심리적으로 분리할지가 플랫폼 신뢰의 핵심 과제가 된다.

## [GitHub Copilot 앱, BYOK(Bring Your Own Key)로 OpenAI·Anthropic·Ollama 등 외부 모델 프로바이더 지원](https://github.blog/changelog/2026-06-23-github-copilot-app-support-for-byok/)

**출시일:** 2026년 6월 23일

GitHub Copilot 앱이 BYOK(Bring Your Own Key) 기능을 추가해 사용자가 자신의 API 키를 사용해 에이전트 세션을 운영할 수 있게 됐다.

### 지원 프로바이더

지원되는 모델 프로바이더는 다음과 같다:

- OpenAI
- Azure OpenAI
- Microsoft Foundry
- Anthropic
- LM Studio
- Ollama
- OpenAI 호환 엔드포인트(내부 게이트웨이 포함)

### 설정 방법

사용자는 **Settings → Model Providers** 메뉴에서 엔드포인트와 API 키를 입력하거나, LM Studio 또는 Ollama의 경우 호스트 주소만 입력해 프로바이더를 추가할 수 있다. 설정이 완료되면 해당 프로바이더의 모델이 Copilot 호스팅 모델과 함께 모델 선택기에 나타난다. "키는 로컬 OS 키체인에 저장되며 UI에서 다시 읽지 않는다"고 GitHub는 명시했다.

### 핵심 기능

BYOK 기능은 세 가지 주요 활용을 가능하게 한다.

**기존 프로바이더 통합:** Azure OpenAI, Anthropic, 자체 호스팅 Ollama, LM Studio, OpenAI 호환 게이트웨이를 현재의 청구·할당량·리전·데이터 처리 계약을 유지하면서 연결할 수 있다.

**모델 유형 조합:** 프론티어 모델과 로컬·자체 호스팅 모델을 함께 사용할 수 있다. 복잡한 작업은 프론티어 모델이, 실행 단계는 로컬 모델이 담당하는 방식으로 분리가 가능하다.

**데이터 경계 유지:** 개인 클라우드 계정, 테넌트, 또는 내부 게이트웨이를 통해 추론을 라우팅해 기업 및 규제 환경의 요구사항을 충족할 수 있다.

### 접근 조건

Copilot Business 또는 Enterprise 플랜에서 GitHub Copilot 앱을 사용하려면 조직 또는 엔터프라이즈 관리자가 정책 설정에서 Copilot CLI를 활성화해야 한다.

### 이 변화의 의미

기업이 코딩 에이전트를 도입할 때 가장 먼저 부딪히는 질문은 "모델이 충분히 똑똑한가"가 아니라 "우리 데이터가 어디로 가고, 누가 비용을 내며, 내부 게이트웨이를 통과할 수 있는가"다. BYOK는 Copilot 앱을 단일 모델 제품에서 여러 모델·리전·청구 정책을 묶는 에이전트 컨트롤 플레인으로 바꾼다. 규제 산업에서 AI 코딩 도구 도입의 병목 하나가 줄어드는 셈이다.

## [Copilot CLI 새 터미널 UI, 일반 제공(GA) 전환](https://github.blog/changelog/2026-06-23-copilot-cli-new-terminal-interface-is-generally-available/)

**출시일:** 2026년 6월 23일

GitHub가 Microsoft Build 2026에서 프리뷰했던 Copilot CLI의 새로운 터미널 인터페이스를 일반 제공(GA)으로 전환했다. 새 UI는 탭 레이아웃, 향상된 도구 설정, 접근성 개선이 포함된다.

### 이슈·PR·Gist 탭 탐색

터미널 세션 상단에 탭이 표시된다. 기본 **Session** 탭과 개인 Gist용 **Gists** 탭 사이를 이동할 수 있으며, GitHub 저장소 내에서 작업할 때는 **Issues** 탭과 **Pull requests** 탭이 추가로 나타난다.

탭 간 이동은 Tab 키, 항목을 프롬프트에 참조할 때는 'c', 브라우저에서 열 때는 'o', GitHub 검색은 '/'를 사용한다. 탭 구성은 설정에서 사용자화할 수 있다.

### 터미널을 벗어나지 않고 도구 설정

도구 설정이 안내식 대화형으로 바뀌었다.

- **MCP 서버:** `/mcp add`로 대화형 양식 입력, `/mcp search`로 GitHub MCP Registry 탐색
- **스킬:** `/skills`로 기능을 켜거나 끄기
- **플러그인:** `/plugin` 명령으로 마켓플레이스·저장소·로컬 설치 접근
- **설정:** `/settings` 다이얼로그로 인라인 구성 변경

### 더 깔끔하고 접근성 높은 인터페이스

재설계된 인터페이스는 테마 인식 시맨틱 색상과 반응형 컴포넌트를 사용한다. `/theme` 명령으로 기본·어두운·고대비·색맹 모드를 선택할 수 있다. 스크린 리더 지원이 자동으로 활성화되며, 아이콘에 레이블이 붙고 애니메이션이 비활성화된다. 좁은 터미널과도 호환된다.

### 업데이트 방법

터미널에서 `copilot update`를 실행해 업데이트할 수 있다. 피드백은 `/feedback` 명령 또는 공개 저장소를 통해 제출할 수 있다.

### 이 변화의 의미

코딩 에이전트는 IDE 안의 대화창에서 끝나지 않고 이슈, PR, Gist, MCP, 스킬, 플러그인을 오가는 운영 도구가 되고 있다. Copilot CLI가 터미널 네이티브 탐색과 도구 설정을 통합한 것은 "코드 작성"보다 "작업 맥락을 빠르게 붙잡아 에이전트에 넘기는 것"이 생산성의 중심이 되고 있다는 신호다.

## [Jules로 중요한 것을 측정하기 — Google의 프로액티브 코딩 에이전트 평가 방법론](https://developers.googleblog.com/measuring-what-matters-with-jules/)

**게재일:** 2026년 6월 22일  
**저자:** Nghi Bui (리서치 사이언티스트), Georgios Evangelopoulos (리서치 사이언티스트), Zack Elliott (소프트웨어 엔지니어)

### 핵심 주장: 에이전트는 '인사이트 정책'으로 평가해야 한다

AI 코딩 에이전트는 반응형 작업 완료 도구에서 떠오르는 위험을 식별하고 진단 인사이트를 제공하는 프로액티브 시스템으로 변화하고 있다. 저자들은 이 변화에 따라 평가 방식도 달라져야 한다고 주장한다. "프로액티브 에이전트는 무엇이 중요한지를 결정하는 능력, 즉 인사이트 정책(insight policy)으로 평가받아야 한다."

### 연구 방법론

연구팀은 Google 내부 코드베이스의 버그 705건(코드 변경 1,178개)을 활용해 평가 프레임워크를 개발했다. 연관된 과거 버그들을 클러스터링해 상위 수준의 개발자 목표를 식별하고, 에이전트가 코드베이스 탐색을 통해 이러한 인사이트를 얼마나 잘 발굴하는지 테스트했다.

평가 과정은 다음과 같다:
1. 과거 버그 fix 이력을 기반으로 관련 버그 클러스터를 구성한다
2. 에이전트가 코드베이스를 능동적으로 탐색하도록 한다
3. 에이전트가 찾아낸 인사이트가 실제 버그 원인 클러스터 내에 얼마나 잘 포함되는지 측정한다

### 주요 결과

- 탐색 1라운드만으로도 에이전트는 평균 **5점 만점에 4.5점**의 관련성 점수를 기록했다
- 탐색 예산을 2라운드에서 3라운드로 늘리면 **Hit@5 정확도가 33%에서 57%로** 향상됐다

즉, 에이전트에게 더 많은 탐색 기회를 줄수록 실제로 중요한 버그 인사이트를 더 잘 잡아낸다는 것이다.

### 향후 계획

연구팀은 공개 GitHub 데이터로 테스트를 확대하고, 이슈 트래커·대화·설계 문서를 포함한 더 풍부한 맥락 스트림을 통합할 계획이다. 연구 논문은 arxiv.org/abs/2605.06717에서 확인할 수 있으며, 추가 정보는 labs.google/code에서 제공된다.

### 이 연구가 왜 중요한가

AI 코딩 도구가 반응형 어시스턴트에서 프로액티브 엔진으로 넘어가면, 기존 SWE-Bench 방식의 과제 해결률만으로는 품질을 설명하기 어렵다. 개발자에게 "이 버그를 고쳐라"고 요청받기 전에 스스로 주목해야 할 버그를 찾아내는 에이전트를 어떻게 평가할 것인가? Google의 답은 단일 태스크 성공률이 아니라 에이전트의 판단력—조용히 있어야 할 때와 개발자에게 알려야 할 때를 구분하는 능력—을 측정하는 것이다.

## [Claude Code의 "Extended Thinking" 출력 텍스트는 진짜 추론이 아니다](https://patrickmccanna.net/the-text-in-claude-codes-extended-thinking-output-is-not-authentic/)

**게재일:** 2026년 6월 22일  
**저자:** Patrick McCanna

Patrick McCanna의 이 글은 Claude Code의 extended thinking 기능의 투명성에 관한 우려를 제기한다.

### 핵심 발견: 로컬 로그에는 암호화된 서명만 있다

McCanna는 로컬에서 세션 로그를 검사하면 실제 추론이 아니라 암호화된 서명만 발견된다는 것을 확인했다. 그의 분석에 따르면 Claude는 추론을 이 서명들에 암호화하며, Anthropic이 복호화 키를 보유한다. 사용자의 기기는 이 키를 받지 못한다.

### API가 반환하는 것의 실체

McCanna가 강조한 핵심 사실:

- API는 **추론 자체가 아니라 추론의 요약**을 반환한다
- 전체 thinking 출력을 보려면 **엔터프라이즈 계약이 필요하다**
- extended thinking 출력은 에이전트 행동을 실제로 구동한 로직이 아니라 모델 사고의 **요약본**이다
- 로컬 추론 로그는 사용자가 접근할 수 없다

### Anthropic 문서에 대한 비판

McCanna는 Anthropic의 문서가 불명확하다고 지적한다. 사용자가 요약본과 실제 thinking 과정의 차이에 관한 중요한 세부 사항을 쉽게 놓칠 수 있다는 것이다. 그는 에이전트 의사결정의 감사 추적(audit trail)을 필요로 하는 모든 사람에게 이 구분이 중요하다고 강조했다.

### 실제적 한계

글의 결론에서 McCanna는 사용자가 로컬 파일에서 실제 추론을 재현할 수 없으며, 입력과 출력을 로깅하는 우회책을 써도 에이전트의 행동을 유발한 추론을 포착하지 못한다고 밝혔다.

### 이 문제가 왜 중요한가

AI 에이전트를 장시간 업무 인프라로 사용할 때, 많은 팀이 무언가 잘못됐을 때 "에이전트가 왜 그 결정을 내렸는가"를 추적할 수 있어야 한다. Claude Code 로그와 extended thinking 요약을 감사 기록으로 착각하면 실제로는 존재하지 않는 가시성을 확보했다고 오해할 수 있다. HN에서도 "보이는 reasoning"과 실제 모델 내부 추론의 경계를 두고 논쟁이 이어진 것은 이 문제가 개발자 커뮤니티에서 실질적인 관심사임을 보여준다.

## [Google ADK와 A2A로 크로스 언어 멀티 에이전트 팀 구성하기](https://developers.googleblog.com/build-cross-language-multi-agent-team-with-google-agent-development-kit-and-a2a/)

**게재일:** 2026년 6월 22일  
**저자:** Shubham Saboo (시니어 AI 프로덕트 매니저), Eric Dong (개발자 관계 엔지니어)

### 개요

이 글은 Python 에이전트와 Go 에이전트가 A2A(Agent-to-Agent) 프로토콜과 Google의 에이전트 개발 키트(ADK)를 사용해 협력하는 계약 컴플라이언스 파이프라인을 구축하는 방법을 보여준다.

### 핵심 문제: 모놀리식 에이전트의 한계

프로덕션 AI 시스템에서 여러 언어로 구축된 에이전트를 조합할 때 세 가지 핵심 문제가 발생한다:

- **맥락 열화(Context degradation):** 도구가 10~15개를 넘어가면 모델이 지시를 놓치고 파라미터를 환각(hallucinate)한다
- **폭발 반경(Blast radius):** 단일 장애가 전체 워크플로를 중단시킨다
- **테스트 가능성(Testability):** 50개 이상의 책임이 얽혀 있으면 단위 테스트가 불가능하다

### 해결책: A2A 프로토콜

A2A(Agent-to-Agent) 프로토콜은 다음 메커니즘으로 언어 간 상호운용성을 가능하게 한다:

- **디스커버리(Discovery):** 에이전트가 Agent Card(JSON 메타데이터, `/.well-known/agent.json` 위치)를 통해 자신의 기능을 광고한다
- **통신(Communication):** HTTP 위의 JSON-RPC 2.0 교환
- **태스크 생명주기(Task lifecycle):** 명확하게 정의된 상태 (submitted → working → completed / failed)

### 구현 세부 사항

이 파이프라인은 세 에이전트로 구성된다:

1. **추출 에이전트(Python/Gemini)** — 계약서에서 정보를 파싱한다
2. **컴플라이언스 에이전트(Go)** — 결정론적 정책 검증을 수행한다
3. **리포트 에이전트(Python/Gemini)** — 요약본을 생성한다

Python 측에서는 `RemoteA2aAgent`를 사용해 Go 서비스를 로컬 하위 에이전트로 래핑한다. Go 측에서는 AI 프레임워크 없이 A2A 프로토콜을 구현하는 표준 HTTP 서버를 사용한다.

### 이 아키텍처의 이점

- 팀이 각자의 전문 분야에 최적화된 언어를 사용할 수 있다
- 추상화 레이어를 통해 통합이 단순해진다
- 수동 검토 폴백(fallback)으로 점진적 성능 저하가 가능하다
- 컴플라이언스 검증을 위한 결정론적 감사 추적이 생성된다

### 이 글이 왜 중요한가

프로덕션 에이전트 시스템은 거대한 프롬프트 하나보다 좁은 책임을 가진 에이전트들과 결정론적 서비스의 조합에 가까워지고 있다. Go로 만든 결정론적 검증기와 Python/Gemini 에이전트를 A2A로 연결하는 이 예제는, 언어·팀·책임의 경계를 유지하면서 멀티 에이전트 파이프라인을 구성하는 실제 패턴을 보여준다. 전체 오픈소스 구현은 GitHub에서 확인할 수 있다.
