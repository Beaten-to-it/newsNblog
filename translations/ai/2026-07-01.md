## [Claude Sonnet 5 출시](https://www.anthropic.com/news/claude-sonnet-5)

Anthropic이 2026년 6월 30일 새 모델 **Claude Sonnet 5**를 공개했습니다. 회사는 이 모델을 "the most agentic Sonnet model yet"(지금까지의 Sonnet 계열 중 가장 에이전트다운 모델)이라고 소개하며, 계획 수립(planning), 도구 사용(tool use), 자율적 작업 수행 능력을 강조합니다. 성능은 상위 모델인 Opus 4.8에 근접하면서도 그보다 낮은 가격대를 유지한다는 것이 핵심 포지셔닝입니다.

가격은 출시 기념(introductory) 단가가 2026년 8월 31일까지 적용됩니다. 100만 input token당 **2달러**, 100만 output token당 **10달러**입니다. 8월 31일 이후의 표준 단가는 input 100만 token당 **3달러**, output 100만 token당 **15달러**로 올라갑니다. 즉 도입 기간이 끝나면 input·output 모두 단가가 1.5배가 되므로, 작업 단위당 비용(cost per completed task)을 다시 계산해야 한다는 점이 실무적으로 중요합니다.

제공 범위는 넓습니다. Free와 Pro 플랜의 **기본 모델**이 되며, Max·Team·Enterprise 사용자도 쓸 수 있습니다. API에서는 모델 식별자 `claude-sonnet-5`로 호출하고, Claude Code와 Claude Platform에서도 바로 사용할 수 있습니다. 회사는 Sonnet 4.6 대비 추론(reasoning), 도구 사용, 코딩, 지식 노동(knowledge work) 전반에서 개선됐다고 밝혔는데, 다만 공개된 발췌 본문에는 구체적인 퍼센트 향상치까지는 명시되어 있지 않습니다.

안전성(safety) 항목도 함께 제시됐습니다. Sonnet 4.6보다 바람직하지 않은 행동(undesirable behaviors)의 전체 발생률이 낮고, 악의적 요청 거부와 prompt injection 공격 저항이 개선됐으며, 환각(hallucination)과 아첨(sycophancy) 비율도 낮아졌다고 합니다. Opus 계열 대비 사이버보안 관련 능력(cybersecurity capability)은 상당히 낮은 수준이며, 출시 시점부터 사이버 안전장치(cyber safeguards)가 기본 활성화된 상태로 제공된다고 밝혔습니다.

한 파트너의 추천사도 인용됐습니다: *"Claude Sonnet 5 gives our agents a strong execution layer for multi-step software engineering work."*(Sonnet 5가 다단계 소프트웨어 엔지니어링 작업에서 우리 에이전트에게 강력한 실행 계층을 제공한다.)

함의는 분명합니다. 2026년 상반기 frontier 경쟁이 "가장 성능 좋은 모델"에서 "에이전트 작업을 대량으로 돌릴 수 있는 가격대"로 옮겨가고 있다는 신호입니다. Sonnet 5가 실제 업무에서 충분히 따라온다면, 기업은 Opus급 고가 모델을 예외적 escalation lane으로 남겨 두고 Sonnet급을 기본 실행 계층으로 두는 라우팅 전략을 택할 수 있습니다.

## [GitHub Copilot에서 Claude Sonnet 5 정식 출시(GA)](https://github.blog/changelog/2026-06-30-claude-sonnet-5-is-generally-available-for-github-copilot/)

같은 날(2026년 6월 30일) GitHub은 **Claude Sonnet 5가 GitHub Copilot에서 정식 출시(generally available)** 됐다고 공지했습니다. 지원되는 Copilot 표면(surface)은 Visual Studio Code, Visual Studio, Copilot CLI, GitHub Copilot cloud agent, GitHub Copilot App, github.com, GitHub Mobile(iOS·Android), JetBrains, Xcode, Eclipse로 폭넓습니다.

이용 가능한 요금제는 Copilot Pro, Pro+, Max, Business, Enterprise 사용자입니다. 단, Business와 Enterprise의 경우 관리자가 직접 켜야 합니다. GitHub의 설명은 다음과 같습니다: *"Copilot Enterprise and Copilot Business plan administrators can enable Claude Sonnet 5 for their organization through the model policy settings in Copilot."*(관리자가 Copilot의 model policy 설정에서 조직 단위로 Sonnet 5를 활성화할 수 있다.)

데이터 정책으로는, GitHub Copilot의 다른 Sonnet 모델과 마찬가지로 Sonnet 5도 **Zero Data Retention(ZDR)** 으로 운영된다고 명시했습니다(*"Like other Sonnet models in GitHub Copilot, Claude Sonnet 5 operates under Zero Data Retention (ZDR)."*). 성능에 대해서는 다양한 코딩 시나리오에서 강한 결과를 보였고, 특히 CLI 형태의 작업(CLI-style tasks)에서 두드러졌으며, prompt-cache 활용도와 latency 경쟁력이 우수하다고 평가했습니다. 과금은 Usage Based Billing 하에서 제공사(provider) 정가가 적용되며, 롤아웃은 점진적으로 진행되어 아직 모든 사용자에게 동시에 열린 상태는 아닙니다.

## [Copilot Agent, JetBrains AI Assistant에서 사용 가능](https://github.blog/changelog/2026-06-30-copilot-agent-is-now-available-in-jetbrains-ai-assistant/)

GitHub은 같은 날 **JetBrains AI Assistant 안에서 GitHub Copilot을 에이전트로 선택**할 수 있게 됐다고 발표했습니다. 핵심 기능은 세 가지입니다.

첫째, 네이티브 agent picker 통합입니다. AI chat의 agent picker를 열어 GitHub Copilot을 현재 대화의 활성 에이전트로 지정할 수 있습니다(*"Open the agent picker in the AI chat and select GitHub Copilot to make it the active agent for the conversation."*). 둘째, 모델 선택입니다. 지원되는 Copilot 모델 중에서 고르고, 추론 깊이(reasoning depth)를 AI chat 안에서 직접 조정해 속도·깊이·비용의 균형을 맞출 수 있습니다. 셋째, 다단계 작업 처리입니다. 다단계 작업을 넘기면 Copilot이 프로젝트를 따라 추론하고, 변경을 제안하고, 명령을 실행하며 사용자와 반복(iterate)합니다.

배경으로, JetBrains와 GitHub은 Agent Client Protocol(ACP)을 통한 기존 연동을 토대로 "더 깊은 통합(a deeper integration)"을 발표했습니다. 향후 계획으로는 Next Edit Suggestions(NES) 지원, 재사용 가능한 기능을 위한 Skills, 도구 전반에 걸친 더 깊은 오케스트레이션이 예고됐습니다.

## [Cost center 단위 사용자별 AI 크레딧 예산](https://github.blog/changelog/2026-06-30-per-user-ai-credit-budgets-available-for-cost-centers/)

GitHub은 Enterprise 관리자가 **cost center 단위로 사용자별(per-user) AI 크레딧 예산**을 설정할 수 있게 했다고 공지했습니다(2026년 6월 30일). 해당 cost center에 속한 모든 구성원에게 동일하게 적용됩니다.

가장 큰 운영상 이점은 예산이 인사 이동을 자동으로 따라간다는 점입니다. 사용자가 직접 또는 enterprise team을 통해 cost center에 합류하면 그 cost center의 사용자별 예산을 그대로 상속하고, 제거되면 별도 재설정 없이 적용이 종료됩니다. 구체적 예시로 GitHub은 다음을 들었습니다: *"You could put your platform engineering team in one cost center at $250 per user and keep everyone else on a $40 universal budget, without setting budgets one user at a time."*(플랫폼 엔지니어링 팀은 사용자당 250달러 cost center에 두고, 나머지는 사용자당 40달러 universal budget으로 두면서, 일일이 개별 설정하지 않아도 된다.)

예산 우선순위(precedence)는 개별 사용자 단위 예산 > cost center 사용자 단위 예산 > universal 예산 순으로 적용됩니다. 한 가지 주목할 차이는, cost center 사용자 단위 예산은 구성원이 소비하는 모든 AI 크레딧을 — 포함된 사용량 풀(included usage pool)과 추가 사용량(additional usage)을 모두 — 집계한다는 점입니다. 따라서 종량 과금이 시작되기 전에 사용을 멈출 수 있어, 추가 사용량만 상한으로 두는 cost center 예산과 구분됩니다. 다만 출시 시점 기준으로는 설정이 REST API로만 가능하며, *"Billing UI support for creating this budget type is coming soon."*(이 예산 유형을 만드는 Billing UI 지원은 곧 제공될 예정.)

## [Claude Science — 과학자용 AI 워크벤치 정식 공개](https://www.anthropic.com/news/claude-science-ai-workbench)

Anthropic이 2026년 6월 30일 과학자를 위한 애플리케이션 **Claude Science**를 베타로 공개했습니다. 연구에 자주 쓰는 도구들을 하나의 환경에 통합해, 문헌 분석부터 다단계 연구 수행, 상세 산출물(artifact) 생성, 그림(figure)과 원고의 반복적 다듬기까지 연구의 전 단계를 한곳에서 처리하도록 설계됐습니다. 모든 산출물에는 그것이 어떻게 만들어졌는지에 대한 **감사 가능한 이력(auditable history)** 이 남습니다.

핵심 수치와 사양은 다음과 같습니다. **60개 이상의 curated skills와 connectors**가 사전 구성되어 있고, genomics, single-cell, proteomics, structural biology, cheminformatics 분야에 맞춰 미리 세팅되어 있습니다. macOS와 Linux에서 로컬로, 또는 SSH/HPC login node를 통해 원격으로 동작합니다. 연결되는 과학 데이터베이스로는 UniProt, PDB, Ensembl, Reactome, ClinVar, ChEMBL, GEO가 명시됐고, NVIDIA의 BioNeMo Agent Toolkit과 BioNeMo 모델(Evo 2, Boltz-2, OpenFold3), 온디맨드 컴퓨트용 Modal, 그리고 PubMed·Jupyter·R이 통합됩니다.

기능 측면에서는 3D 단백질 구조, genome browser track, 화학 구조, 그림, 원고 같은 풍부한 과학 산출물을 재현 가능한 코드와 함께 네이티브로 렌더링합니다. 컴퓨트 관리 측면에서는 SSH나 Modal 계정을 통해 HPC 클러스터에 job을 제출하고, 단일 GPU에서 수백 개 규모까지 확장하며, 노트북·Linux 머신·HPC login node 환경을 관리합니다. 특히 **Review Agent**가 산출물을 점검해 잘못된 인용(incorrect citations), 추적 불가능한 수치(untraceable numbers), 코드와 어긋난 그림(figures mismatched to underlying code)을 표시하고 스스로 교정합니다. 세션을 fork해 원래 흐름을 잃지 않고 여러 접근을 비교할 수도 있습니다.

원문 인용 일부는 다음과 같습니다: *"Scientific research is often tedious. Researchers must work across dozens of databases, each with their own schema..."* / *"Every output carries an auditable history of how it was made, so you can validate and reproduce the results."*(모든 산출물에는 그것이 만들어진 감사 가능한 이력이 담겨 결과를 검증하고 재현할 수 있다.) / *"When it generates a figure, Claude Science includes the exact code and environment that produced it..."*(그림을 생성하면 그 그림을 만든 정확한 코드와 환경을 함께 포함한다.)

실제 사용 사례도 제시됐습니다. **Manifold Bio**는 조직 표적 의약품을 위한 약물 표적(drug target)을 추려내며 표면 발현(surface expression), 트래피킹(trafficking), 안전성을 평가했습니다. **Allen Institute의 Jérôme Lecoq**은 약 20개의 커스텀 skill로 multi-agent "computational review template"을 구축해, 리뷰 작성 시간을 약 2년에서 100쪽이 넘는 리뷰를 10편 이상 생산하는 수준으로 단축했습니다. **UCSF Brain Tumor Center의 Stephen Francis**는 glioma 분자역학(molecular epidemiology) 분석을 기존의 약 10분의 1 시간으로 가속하고 결과를 독립적으로 검증했다고 밝혔습니다.

지원 프로그램(AI for Science)도 함께 공지됐습니다. 최대 **50개**의 Claude Science AI for Science 프로젝트에 프로젝트당 최대 **3만 달러($30,000)** 의 크레딧을 제공하고, Modal이 선정 프로젝트에 최대 **2,000달러($2,000)** 의 컴퓨트 크레딧을 지원합니다. 지원 마감은 **7월 15일**, 선정 통보는 **7월 31일**, 프로젝트 기간은 **9월 1일부터 12월 1일**까지입니다.

함의로는, AI for Science가 챗봇의 "논문 요약"을 넘어 코드·환경·데이터 출처·reviewer agent·감사 가능한 이력이 함께 남는 업무 환경으로 이동하고 있음을 보여줍니다. 과학 조직은 모델 성능만이 아니라 "몇 달 뒤 재실행이 가능한가"를 평가 기준으로 삼아야 한다는 점이 핵심입니다.

## [Gemini Omni Flash와 Nano Banana 2 Lite로 속도·비용 경쟁력 확보](https://cloud.google.com/blog/products/ai-machine-learning/nano-banana-2-lite-and-gemini-omni-flash-available/)

**Nano Banana 2 Lite**(정식 명칭 Gemini 3.1 Flash-Lite Image)는 정식 출시(GA) 상태로, Nano Banana 모델 패밀리 안에서 가장 빠르고 비용 효율적인 이미지 생성·편집 모델입니다. 핵심 사양으로 **이미지 한 장을 최소 4초(four seconds)에 생성**하며, 구형 Nano Banana(Gemini 2.5 Flash Image) 대비 크게 개선됐습니다. 정확한 맥락의 장면 초안, 대략적인 데이터 시각화, 위치 특화 목업을 그릴 수 있는 world knowledge, 여러 차례의 빠른 생성에서도 캐릭터 정체성과 객체 충실도를 유지하는 character consistency, 빠른 생성물 안에 읽을 수 있는 텍스트를 렌더링하는 기능을 갖췄습니다. 다만 이미지 편집은 생성보다 응답 시간이 약간 더 길 수 있다고 안내됐습니다.

## [X, 호스티드 MCP 서버 공개](https://docs.x.com/tools/mcp)

브리핑에서는 TechCrunch가 X 측 확인으로 이 MCP가 Write API endpoint를 제공하지 않는다고 전했다고 정리했으나, 공식 문서 본문에는 쓰기(write) 작업이 명시적으로 지원된다고 적혀 있습니다: *"Required for writes (bookmarks, Articles)"* 이며 *"Writes count against rate limits."*(북마크·Articles 같은 쓰기에는 별도 권한이 필요하고, 쓰기는 rate limit에 집계된다.) 즉 자동 게시(트윗 작성)용 일반 write endpoint와, 북마크·Articles 같은 특정 쓰기 동작은 구분해서 이해할 필요가 있습니다.

## [Claude Code가 요청을 스테가노그래피로 표시한다는 주장](https://thereallo.dev/blog/claude-code-prompt-steganography)

> (원문 접근 제한: thereallo.dev 블로그 본문이 HTTP 403으로 직접 열리지 않았습니다. 아래는 이 글을 다룬 Hacker News 토론(https://news.ycombinator.com/item?id=48734373, 1183 points·310 comments, 제출자 kirushik)에서 확인된 범위로 정리하며, 모두 "주장" 단계임을 전제합니다.)

한 개발자가 Claude Code의 바이너리를 분석해, 클라이언트가 요청에 스테가노그래피(steganography) 방식의 표지(marker)를 심는다고 주장했습니다. HN 토론에서 정리된 기술적 주장은 다음과 같습니다.

- **타임존 감지**: 시스템 컨텍스트의 날짜 문자열 포맷이 사용자 타임존에 따라 달라진다.
- **도메인 블랙리스트**: 호스트명을 약 200개 이상의 도메인 목록과 대조하며, 여기에 Baidu, Alibaba, ByteDance, DeepSeek 등 중국계 기술 기업과 리셀러 서비스가 포함된다(예시로 거론된 도메인: deepseek, moonshot, minimax, baichuan, stepfun, zhipu 등).
- **구현 방식**: JavaScript 코드가 환경 변수와 시스템 정보를 확인해 시스템 프롬프트에 표지를 삽입한다.
- **표지 유형**: 감지 신호를 인코딩하는, 눈에 보이지 않는 문자(Unicode steganography).

토론은 두 갈래로 갈렸습니다. **핑거프린팅을 옹호하는 쪽**은 Anthropic이 해외 연구소가 자사 모델을 distillation한다고 공개적으로 밝혀 온 만큼 워터마킹이 합리적인 anti-abuse 조치이고, 의심스러운 패턴만 표시하므로 일반 개발자에게는 영향이 없으며, 사기 탐지에 흔히 쓰이는 기법이라고 봤습니다(한 사용자 edude03: *"[Steganography isn't] security by obscurity...keeping the private thing private is part of the security model"*). **투명성·신뢰를 우려하는 쪽**은 release note나 개인정보 정책에 고지되지 않았다는 점, 표지가 더 넓은 감시나 출력 품질 저하로 확대될 위험(scope creep), 정상적인 프록시·다계정 워크플로·자격증명 필터링이 오탐(false positive)으로 불이익을 받을 수 있다는 점을 지적했습니다(한 사용자 civet_java: *"That they choose to implement it by fingerprinting my access patterns without first disclosing is where they shit the bed"*).

함의는, 코딩 에이전트가 로컬 파일시스템과 shell 권한을 넓게 받는 만큼, 모델 호출 전 컨텍스트에 무엇이 들어가는지, 클라이언트가 어떤 환경 신호를 수집하는지, 우회 방지(anti-abuse)와 사용자 고지가 어디서 균형을 잡아야 하는지가 중요한 운영·보안 이슈가 됐다는 점입니다. 특히 proxy·gateway·사내 모델 라우터를 쓰는 팀은 클라이언트 업데이트가 요청 컨텍스트를 어떻게 바꾸는지 확인할 필요가 있습니다.
