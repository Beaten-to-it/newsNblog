## [OpenAI와 Broadcom, LLM에 최적화된 추론 칩 'Jalapeno' 공개](https://openai.com/index/openai-broadcom-jalapeno-inference-chip/)

**(원문 접근 제한: openai.com 공식 발표 페이지와 Broadcom 보도자료는 각각 HTTP 403, 응답 시간 초과로 본문을 읽지 못했습니다. 아래 내용은 같은 사안을 보도한 TechCrunch 기사(직접 확인)와 브리핑에 이미 담긴 정보의 범위 안에서 정리했습니다.)**

OpenAI가 Broadcom과 함께 만든 첫 자체 추론(inference) 프로세서 'Jalapeno'를 공개했습니다. TechCrunch 보도에 따르면 두 회사의 협력은 이미 2025년 10월에 공식 발표됐고, 이번에 그 결과물인 맞춤형 칩의 실체가 처음으로 드러난 것입니다. 칩의 이름은 매운 고추 '할라피뇨(Jalapeño)'를 그대로 가져왔습니다.

이 칩은 '학습(pre-training)'이 아니라 '추론'에 특화돼 있습니다. 추론이란 이미 학습이 끝난 AI 모델을 사용자 명령에 따라 실제로 돌리는 단계를 말합니다. OpenAI는 특히 실시간 코딩 모델 같은 워크로드를 낮은 운영 비용으로 돌리는 데 칩의 효율이 좋다는 점을 강조했고, 사전학습처럼 연산 부담이 큰 작업은 앞으로도 Nvidia 하드웨어에 의존할 가능성이 높다고 TechCrunch는 전했습니다.

성능과 관련해 OpenAI는 초기 테스트에서 "현재 최고 수준의 대안 대비 전력당 성능(performance-per-watt)이 크게 낫다"고 밝혔습니다. 흥미로운 대목은 OpenAI 자신의 AI 모델들이 이 칩 개발 과정에 활용됐다는 점입니다.

전략적으로 이 발표는 OpenAI가 오랫동안 안고 있던 Nvidia GPU 의존 문제를 정면으로 겨냥합니다. 이미 Google과 Amazon도 비슷하게 자체 AI 가속기를 개발해 왔습니다. Greg Brockman OpenAI 사장은 회사가 "충분히 지원받지 못하던 특정 워크로드(specific workloads that are underserved)"를 찾아 "가능한 것을 가속(accelerate what's possible)"하려 했다고 말했습니다.

OpenAI는 이를 단순한 칩 한 개가 아니라 인프라 전반을 직접 통제하려는 수직통합의 일부로 설명합니다. "OpenAI는 스택 전반에 걸쳐 일한다(OpenAI operates across the stack)"며 칩 아키텍처, 커널, 메모리 시스템, 네트워킹, 배포 시스템까지 함께 설계해 성능·안정성·경제성을 최적화한다는 것입니다.

배포 시점에 대해 TechCrunch는 아직 테스트 단계이며 광범위한 배포 일정은 명시되지 않았다고 적었습니다. 다만 브리핑에 따르면 OpenAI는 이 칩을 GPT-5.3-Codex-Spark 같은 실제 워크로드로 'production target frequency와 power'에서 실험실 가동 중이라 설명했고, 연말 배포를 목표로 하는 다세대(multi-generation) 컴퓨트 플랫폼의 첫 단계로 자리매김했습니다. 제조 파트너로는 Celestica가 함께 거론됩니다. (단, 이 마지막 세부 사항들은 차단된 OpenAI 원문에 근거한 브리핑 요약이며, 1차 출처를 직접 확인하지는 못했습니다.)

### 함의

프런티어 모델 회사의 비용 구조가 점점 '학습'이 아니라 '추론'에 좌우되고 있다는 신호입니다. 경쟁의 축이 모델 성능 그 자체에서 칩·커널·메모리·네트워크·데이터센터까지 묶는 풀스택 운영 효율로 이동하고 있음을 보여줍니다.

## [Gemini Omni 공개](https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-omni/)

Google이 추론 능력과 창작 생성 능력을 결합한 새 모델 패밀리 'Gemini Omni'를 공개했습니다. 글은 Google DeepMind CTO이자 Google 수석 AI 아키텍트인 Koray Kavukcuoglu가 작성했습니다.

첫 모델은 **Gemini Omni Flash**입니다. 이미지·오디오·비디오·텍스트를 다양하게 조합해 입력으로 받고, 그 입력들로부터 영상을 생성합니다. 핵심은 단발성 생성이 아니라 '대화형 편집'입니다. 사용자는 자연어로 대화하듯 영상을 고칠 수 있고, 모델은 이전 지시를 이어받아 장면·동작·스타일·카메라 각도를 수정합니다. 여러 번 편집해도 등장인물의 일관성, 물리적 사실성, 장면 연속성이 유지된다고 Google은 설명합니다. 모델이 중력, 운동 에너지, 유체 역학 같은 물리 현상을 이해한다는 점도 강조됐습니다.

또한 Omni는 단순한 패턴 매칭을 넘어, 역사·과학·문화적 맥락에 대한 Gemini의 실제 세계 지식에 영상 생성을 '접지(ground)'시킨다고 밝혔습니다. 언어, 이미지, 의미를 연결한다는 것입니다. 참조(reference) 입력도 다양하게 받습니다. 이미지·텍스트·영상·음성 오디오를 레퍼런스로 넣을 수 있고, 이를 하나의 일관된 결과물로 섞으며, 레퍼런스로부터 포즈·동작·스타일을 전이(transfer)할 수 있습니다.

### 배포 경로

- **Gemini 앱**: Google AI Plus·Pro·Ultra 구독자 대상, 전 세계
- **Google Flow**: 모든 티어
- **YouTube Shorts·YouTube Create App**: 무료 제공, 이번 주 출시
- 개발자·기업용 API: 향후 몇 주 내 제공 예정

### 책임 있는 AI 장치

생성된 모든 영상에는 사람이 알아차리기 어려운 **SynthID** 디지털 워터마크가 들어가 검증에 쓰입니다. **아바타(avatar)** 기능은 사용자가 자신의 목소리·외형으로 영상을 만들 수 있게 합니다. 음성·오디오 편집은 아직 '책임 있는 개발(responsible development)' 차원의 테스트 단계에 있고, 오남용을 막기 위한 정책을 둔다고 밝혔습니다.

### 함의

영상 생성은 한 번의 프롬프트보다 반복 편집과 레퍼런스 유지가 더 중요합니다. Omni가 대화형 편집·입력 조합·SynthID 검증을 한데 묶은 것은 생성형 영상이 '데모'에서 '제품 워크플로'로 넘어가는 신호로 읽힙니다. 동시에 출처 표시(SynthID)와 인물 권리 보호(자신의 voice/avatar 중심으로 시작)가 품질 못지않은 관건임을 보여줍니다.

## [Gemini 3.5 Flash에 computer use 도입](https://blog.google/innovation-and-ai/models-and-research/gemini-models/introducing-computer-use-gemini-3-5-flash/)

Google이 'computer use(컴퓨터 사용)'를 Gemini 3.5 Flash의 내장 도구(built-in tool)로 통합했다고 2026년 6월 24일 발표했습니다. 이 기능은 이전에는 독립된 Gemini 2.5 모델 형태로만 제공됐는데, 이제 Flash에 네이티브로 들어가면서 에이전트 작업 성능이 개선됐습니다.

모델은 "브라우저, 모바일, 데스크톱 환경을 보고, 추론하고, 행동(see, reason and take action)"할 수 있습니다. 특히 장기 호흡(long-horizon)의 작업과 기업용 자동화 작업에 강하다고 소개됐습니다. 예로는 소프트웨어를 지속적으로 테스트하는 일, 여러 전문 애플리케이션을 넘나드는 지식 노동이 거론됩니다.

### 제공 경로

- Gemini API
- Gemini Enterprise Agent Platform

### 안전장치

기업용 선택형 안전장치 두 가지가 함께 공개됐습니다.

1. 민감하거나 되돌리기 어려운 동작에 대해 사용자의 명시적 확인(confirmation)을 요구
2. 간접 프롬프트 주입(indirect prompt injection)이 감지되면 작업을 자동으로 중단

Google은 프롬프트 주입 위험을 줄이기 위해 표적 적대 훈련(targeted adversarial training)을 적용했다고 밝혔습니다. 그러면서 이 도구들을 안전한 샌드박싱, 사람이 개입하는 검증(human-in-the-loop), 엄격한 접근 통제와 함께 쓰는 '심층 방어(defense-in-depth)' 접근을 권고했습니다.

성능과 관련해서는 OSWorld 벤치마크에서 경쟁력 있는 수치를 보였다고 언급됐으나, 발췌된 본문에는 구체적인 수치까지는 명시되지 않았습니다. 시작용으로는 Browserbase가 호스팅하는 데모 환경, 그리고 Gemini API·Enterprise Agent Platform을 통한 참조 구현과 문서가 제공됩니다.

### 함의

computer-use 에이전트는 실제 업무 앱을 클릭하고 입력하기 때문에 성공률보다 권한·샌드박스·확인 절차가 중요합니다. Google이 모델 기능과 기업용 안전장치를 함께 내놓은 점은, 화면 조작형 에이전트가 실험 단계를 넘어 운영 설계의 대상으로 옮겨가고 있음을 보여줍니다.

## [Claude Tag 공개](https://www.anthropic.com/news/introducing-claude-tag)

Anthropic이 2026년 6월 23일 'Claude Tag'를 공개했습니다. Slack 채널에서 @Claude를 태그해 하나의 팀 동료처럼 업무를 위임하는 통합 기능입니다. 사용자가 일을 맡기면 Claude는 관리자가 허용한 채널·도구·데이터·코드베이스에 접근해 일을 처리합니다. 현재 Claude Enterprise·Team 고객 대상 **베타**로 제공되며, **Claude Opus 4.8** 모델 위에서 작동합니다.

가장 눈에 띄는 수치는 Anthropic이 밝힌 **자사 제품팀 코드의 65%가 내부 버전 Claude Tag로 생성된다**는 대목입니다. The Decoder는 이를 "Claude Tag가 Anthropic의 제품팀 코드의 65%를 생성한다"는 회사 측 주장으로 전했습니다.

### 작동 방식

구성원이 @Claude를 태그해 작업을 요청하면, Claude는 일을 여러 단계로 쪼개 사용 가능한 도구로 순차 실행하고, 결과를 Slack 스레드에 보고합니다. 핵심 특징은 다음과 같습니다.

- **멀티플레이어·공유 맥락**: 채널마다 Claude는 하나입니다. 모든 구성원이 진행 중인 작업을 함께 보고 대화를 이어갈 수 있습니다.
- **시간에 따른 학습**: Claude는 자신이 속한 채널에서 관련 정보를 기억하며 맥락을 쌓고, 연결된 데이터 소스로부터 자동으로 학습할 수 있습니다.
- **능동적 행동(ambient mode)**: 주변 동작이 켜지면 Claude가 관련 정보를 먼저 끌어올리고, 해결되지 않은 스레드를 자율적으로 후속 처리합니다.
- **비동기 작업**: 작업이 독립적으로 실행되며, Claude가 미래의 작업을 스스로 예약할 수 있습니다.
- **다이렉트 메시지**: 개인의 도구 설정을 사용해 비공개로 응답합니다.

### 관리·통제

- 시스템 관리자가 채널별로 Claude의 도구·데이터 접근 권한을 지정합니다.
- 용도별로 분리된 Claude 정체성(identity)을 둘 수 있고, 각 메모리는 격리됩니다. 예를 들어 영업용 Claude와 엔지니어링용 Claude는 데이터나 메모리를 공유하지 않습니다.
- 조직 단위·채널 단위로 토큰 지출 상한(spend cap)을 설정할 수 있습니다.
- Claude가 수행한 모든 작업과 요청자를 보여주는 전체 action log가 제공됩니다.
- 채널 범위로 접근을 한정해 도메인 간 정보 유출을 막습니다.

### 도입과 마이그레이션

도입은 4단계입니다. (1) Claude Tag를 Slack 워크스페이스와 연결, (2) Claude에 도구 접근 권한 부여, (3) 조직의 월간 지출 한도 설정, (4) private 채널에서 테스트. 기존 'Claude in Slack' 앱을 대체하며, 30일간의 opt-in(전환) 기간이 주어집니다. 자격이 되는 조직에는 초기 출시 크레딧이 지급됩니다.

The Decoder는 엔지니어링 외에도 지표 추적, 지원 티켓 처리, 버그 진단 같은 용도로 쓰인다고 덧붙였습니다.

### 함의

개인 어시스턴트보다 팀 채널형 에이전트가 더 민감합니다. 한 채널에 모인 사람·대화·권한·도구가 모두 섞이기 때문입니다. Claude Tag의 성패는 모델 성능보다 채널별 메모리 경계, 도구 권한, action log, 월간 지출 한도 같은 운영 기능에 달려 있습니다.

## [Free·Student 플랜의 모델 선택 방식 변경](https://github.blog/changelog/2026-06-24-changes-to-model-selection-for-free-and-student-plans/)

GitHub가 2026년 6월 24일, Copilot Free·Student 플랜 사용자를 위한 모델 선택 경험을 바꿨습니다.

핵심은 **자동 모델 선택(auto model selection)을 기본이자 유일한 모델 선택 방식**으로 만든 것입니다. 자동 모드는 작업별로 가장 적합한 모델을 동적으로 골라, 사용자가 수동으로 선택할 필요를 없애면서도 플랜 한도 안에서 여러 모델 패밀리를 쓸 수 있게 합니다.

이와 함께 Microsoft가 출시한 모델에 붙던 **`(Preview)` 라벨도 제거**됩니다. GitHub는 자동 라우팅이 모델 선택을 관리하고 백엔드 개선이 계속되는 상황에서, 이 라벨이 더는 사용자 판단에 필요하지 않다고 설명했습니다.

부수적으로, 사용자는 자동 모드에서 지원되는 모델에 관한 문서를 참고할 수 있고, 피드백을 나눌 커뮤니티 논의 포럼도 마련돼 있습니다. 변경의 취지는 결정의 복잡성을 줄여 전체 사용자 경험을 단순화하는 데 있습니다. 최근 관련 업데이트로는 Copilot CLI의 새 터미널 인터페이스(6월 23일), JetBrains IDE 개선(6월 22일), 플랫폼 전반의 모델 제공 확대(6월 18일)가 함께 언급됐습니다.

### 함의

개발자 도구에서 모델 선택은 점점 UI 옵션이 아니라 라우팅 정책이 되고 있습니다. 무료·학생 사용자부터 수동 선택을 줄이는 것은, GitHub가 Copilot을 '모델 목록'보다 '작업 단위 자동 제어면'으로 운영하려는 방향을 보여줍니다.

## [TypeScript 7.0 RC 공개](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0-rc/)

Microsoft가 2026년 6월 18일 TypeScript 7.0 릴리스 후보(RC)를 발표했습니다. 가장 큰 변화는 컴파일러를 기존의 TypeScript/JavaScript에서 **Go 언어로 완전히 새로 작성(rewrite)**한 것이며, 이를 통해 상당한 성능 향상을 얻었습니다.

### 성능과 기술 세부

- TypeScript 6.0 대비 **약 10배 빠른** 속도를 냅니다. 네이티브 코드 속도와 공유 메모리 병렬화 덕분입니다.
- 기존의 자체 부트스트랩(self-hosted) TypeScript 코드베이스가 아니라 완전히 새로운 Go 기반 토대 위에 만들어졌습니다.
- 다만 타입체킹 '로직' 자체는 6.0과 구조적으로 동일하게 유지해 호환성을 지켰습니다.
- Microsoft 및 외부 회사들의 수백만 줄 규모 코드베이스에서 이미 테스트됐습니다.

설치는 `npm install -D typescript@rc`로 하고, `npx tsc --version`으로 버전을 확인할 수 있습니다.

### 주요 기능 변화

- **병렬화**: `--checkers` 플래그로 타입체킹 워커 수를 조정합니다(기본 4). `--builders` 플래그는 프로젝트 참조의 병렬 빌드를, `--singleThreaded` 플래그는 단일 스레드 동작을 강제합니다.
- **watch 모드**: Parcel 번들러의 파일 워처를 기반으로 다시 구현해, 자원 소비를 줄이면서 크로스플랫폼에서 효율적으로 동작합니다.
- **설정 기본값 변경**: `strict`가 기본 `true`, `module`이 기본 `esnext`, `target`이 현행 안정 ECMAScript, `stableTypeOrdering`이 `true`(비활성화 불가)로 바뀌었습니다.
- **제거된 항목(deprecations)**: `target: es5`, `downlevelIteration`, `moduleResolution: node/node10`, 구형 모듈 시스템(amd·umd·systemjs), `baseUrl` 등이 더 이상 지원되지 않습니다.

### 에디터 경험

VS Code용 'TypeScript Native Preview' 확장이 매끄러운 통합을 제공합니다. 언어 서버 프로토콜(LSP) 기반이며, 실패하는 언어 서버 명령이 6.0 대비 **20배 줄었습니다**. 자동 import, inlay hint, 시맨틱 하이라이팅 등을 지원합니다.

### 병행 사용과 일정

`@typescript/typescript6` 패키지를 통해 7.0과 6.0을 나란히 돌릴 수 있습니다(`npm install -D typescript@npm:@typescript/typescript6`). 정식 릴리스는 RC 발표 후 한 달 이내로 예상되며, TypeScript 7.1은 안정적인 프로그래매틱 API 기능에 집중할 예정입니다.

### 함의

에이전트가 타입체크와 빌드를 더 자주 돌릴수록 도구 속도는 곧 반복(iteration) 비용이 됩니다. 네이티브 컴파일러와 LSP 기반 편집 경험은 대형 TS 코드베이스의 빌드 시간·메모리 사용을 줄여 AI 코딩 워크플로에도 직접적인 영향을 줍니다.
