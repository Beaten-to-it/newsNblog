## [GitHub Copilot에 Claude Opus 4.8 fast mode가 프리뷰로 추가됐습니다](https://github.blog/changelog/2026-06-29-claude-opus-4-8-fast-mode-is-now-in-preview-for-github-copilot/)

2026년 6월 29일, GitHub Copilot이 Claude Opus 4.8의 'fast mode'를 프리뷰(preview)로 추가했습니다. GitHub는 이 모드를 두고 "Claude Opus 4.8과 동일한 지능(intelligence)을 유지하면서 출력 토큰(output token) 속도를 크게 끌어올린(significantly faster output token speeds) 모드"라고 설명합니다. 즉 모델의 똑똑함은 그대로 두고, 답이 흘러나오는 속도를 높여 대화형 코딩과 에이전트형 워크플로(agentic workflow)처럼 속도가 곧 사용 경험을 좌우하는 작업에 최적화했다는 것입니다.

적용 범위는 넓습니다. Visual Studio Code, Visual Studio, Copilot CLI, GitHub Copilot cloud agent, GitHub Copilot 앱, github.com 웹, GitHub Mobile(iOS·Android), JetBrains, Xcode, Eclipse 등 주요 클라이언트에 걸쳐 점진적으로(gradual rollout) 배포되며, 시간이 지나면서 가용 범위가 늘어납니다.

요금제 기준으로는 Copilot Pro+, Max, Business, Enterprise 사용자가 쓸 수 있습니다. 과금은 사용량 기반 과금(Usage Based Billing)으로 공급자 정가(provider list pricing)에 따라 청구되며, fast mode는 이전 세대의 fast mode보다는 저렴하지만 표준 Claude Opus 4.8보다는 비싸게 책정됩니다.

특히 운영 관점에서 중요한 점은 정책 설정입니다. Copilot Business와 Enterprise 관리자에게 fast mode 정책은 '기본 꺼짐(off by default)' 상태로, 관리자가 Copilot 설정에서 직접 켜야만 사용할 수 있습니다. 빠른 속도를 전사 기본값으로 둘지, 비용을 고려해 대화형 코딩 일부 시나리오에만 허용할지를 조직이 명시적으로 정해야 한다는 뜻입니다.

## [Micro-Agent: 모델 API 안에서의 협업으로 프런티어 모델을 능가하기](https://vllm.ai/blog/2026-06-29-micro-agent-frontier-models)

vLLM Semantic Router 팀이 'micro-agent'라는 개념을 공개했습니다. 핵심 발상은 하나의 통합된 모델 API 표면(single unified model API surface)은 그대로 유지하면서, 그 API 호출 한 번을 "서빙 레이어(serving layer) 내부의 제한된 협업(bounded collaboration)"으로 바꾸는 것입니다. 애플리케이션이 직접 에이전트 그래프를 짜지 않아도, 서빙 계층이 여러 모델의 협업·판단·합성을 대신 처리합니다.

### 다섯 가지 루퍼(Looper) 알고리즘

이 런타임은 작업 성격에 맞는 협업 패턴을 고를 수 있도록 다섯 가지 알고리즘을 제공합니다.

- **Confidence Loop(신뢰도 루프)**: 순차적 에스컬레이션 방식으로, 토큰 단위 확률이나 검증 시스템에서 나온 신뢰도 점수를 평가해 임계값을 못 넘길 때만 더 강한 모델로 올립니다.
- **Ratings Loop(평점 루프)**: `max_concurrent` 한도 안에서 병렬 실행하고, 성공한 응답들을 평점 가중치(rating-aware weighting)로 종합합니다.
- **ReMoM(Repeated Mixture-of-Models)**: 여러 추론 시도를 펼친 뒤 최소 성공 정족수(minimum-success quorum)를 기다렸다가, 출력 계약(output contract)을 보존하면서 결과를 합성합니다.
- **Fusion(융합)**: '패널–심판–최종(panel-judge-final)' 구조로, 독립적인 모델 응답들을 증거로 삼아 심판 모델이 분석합니다. 이때 모델 간 '불일치를 잡음이 아니라 신호로 취급(disagreement as signal)'하는 것이 특징입니다.
- **Workflows(워크플로)**: planner·patcher·verifier·finalizer 같은 역할 기반 시스템으로, "최대 단계 수, 최대 병렬성, 타임아웃, 오류 정책(max steps, max parallelism, timeouts, and error policy)"이라는 엄격한 거버넌스 아래 돌아갑니다.

여기에 더해 'Auto Recipes'는 요청 특성에 따라 내부적으로 적절한 협업 패턴을 자동 선택하면서도 `vllm-sr/auto`라는 단일 정체성을 유지합니다.

### "최선의 루프는 작업의 모양을 닮는다"

글의 기술 철학은 "The best loop is task-shaped"라는 문장에 담겨 있습니다. 벤치마크마다 필요한 패턴이 다르다는 것입니다. 예컨대 GPQA-Diamond는 ReMoM으로 답을 엄격히 보존해야 하고, LiveCodeBench는 코드 중심 검증 루프가 필요하며, 추론 과제는 Fusion의 불일치 분석에서 이득을 본다고 설명합니다.

### 벤치마크 결과

| 벤치마크 | VSR Closed | 비교 대상 |
|---|---|---|
| LiveCodeBench | 92.6 | Fugu Ultra 92.0, GPT-5.5 90.7 |
| GPQA-Diamond | 96.0 | Fugu Ultra 95.5, Gemini 3.1 Pro 94.3 |
| Humanity's Last Exam | 50.0 | Fugu Ultra 50.0 |

오픈 모델과 클로즈드 모델을 섞은 VSR Hybrid는 Humanity's Last Exam에서 47.1을 기록해, GLM-5.2의 40.5와 경쟁 모델의 41.4를 앞섰습니다.

핵심 주장은 이제 '프런티어 모델(frontier model)'이라는 말이 단일 체크포인트(checkpoint)뿐 아니라 시스템 경계(system boundary)까지 포괄한다는 것입니다. 뛰어난 성능은 개별 모델 하나에서가 아니라, "언제 비용을 아끼고, 언제 안전을 강제하며, 언제 한계까지 밀어붙이고… 언제 하나의 요청을 작고 규율 있는 팀으로 바꿀지"를 아는 '더 나은 라우터(better routers)'에서 나온다고 말합니다. 이 작업은 vLLM Semantic Router 팀이 주도했으며, MBZUAI·McGill University·Mila·Agentic Intelligence Lab의 협력자들, 그리고 Huamin Chen·Yincheng Ren 등이 기여자로 명시돼 있습니다.

## [유럽의 AI 인력 기회 지도 그리기 (Mapping Europe's AI Workforce Opportunity)](https://openai.com/index/mapping-ai-jobs-transition-eu/)

> (원문 접근 제한: openai.com 본문은 HTTP 403으로 직접 읽지 못했습니다. 아래 내용은 OpenAI 공식 보고서 요지를 정리한 공개 검색 결과와 [보고서 PDF](https://cdn.openai.com/pdf/the-ai-jobs-transition-framework_report.pdf)에서 확인된 범위로만 정리했습니다.)

OpenAI Economic Research가 EU 노동시장에 맞춰 'AI Jobs Transition Framework'(AI 일자리 전환 프레임워크)의 유럽판을 공개했습니다. 이 프레임워크는 유럽연합 공식 직업 분류 체계인 ESCO(European Skills, Competences, Qualifications and Occupations)와 Eurostat(유럽 통계청) 고용 데이터를 사용해, AI 역량이 EU 회원국별로 어떤 단기 직업 변화로 이어질 수 있는지를 살핍니다.

프레임워크는 EU 일자리를 네 가지 전환 유형으로 나눕니다. AI와 함께 성장할 수 있는 직업, 자동화 잠재력이 높은 직업, 업무가 재조직될 직업, 그리고 단기 변화가 작은 직업입니다. 공개된 수치를 보면 전체 고용 중 약 **18%가 자동화 위험이 높은 직업, 24%가 AI 통합으로 재조직될 직업, 12%가 AI와 함께 성장할 직업, 46%가 단기 변화가 작을 것으로 예상되는 직업**으로 분류됩니다.

주요 발견 중 하나는 미국과 비교했을 때 EU는 단기 자동화 잠재력이 높은 직업의 고용 비중이 더 작다는 점입니다. OpenAI는 이 유럽판 프레임워크를 '준비를 위한 지도(a map for preparation)'로 규정합니다. AI 역량이 특정 직업과 특정 제도적 환경에서 어떻게 경제적 변화로 바뀌는지를 더 유용하게 질문하기 위한 도구이며, 더 나은 근거가 노동자·기업·정책 입안자에게 준비할 시간을 더 준다는 취지입니다.

## [Gemini의 개인화 AI 이미지 생성, 이제 미국 무료 사용자에게도 무료](https://techcrunch.com/2026/06/29/geminis-personalized-ai-image-generation-is-now-free-for-u-s-users/)

Google이 Gemini 앱의 'Nano Banana' 기반 개인화 이미지 생성 기능을 미국의 적격 무료 사용자(eligible free users)에게도 확대했습니다. 그동안 이 기능은 유료 구독자(Plus·Pro·Ultra 등급)만 쓸 수 있었는데, 이번 확대로 그 제한이 풀렸습니다.

작동 방식의 핵심은 사용자가 일일이 자세히 설명하지 않아도 관심사를 반영한 이미지를 만들어 준다는 점입니다. TechCrunch가 든 예시로는 "나와 커피·베이킹 같은 내가 좋아하는 것들로 일러스트를 만들어줘"처럼 길게 쓰는 대신, "나와 내가 좋아하는 것들로 일러스트를 만들어줘"라고만 해도 됩니다. Gemini가 연결된 Google 서비스, 즉 Gmail, Google Photos, YouTube, Google Search의 데이터를 활용해 사용자의 취향을 파악하기 때문입니다. 특히 Google Photos에서 실제 사용자 사진을 끌어와 별도로 업로드하지 않아도 됩니다.

개인정보 측면에서는 이 기능이 옵트인(opt-in)이라는 점이 강조됩니다. 사용자가 Gemini에 어떤 앱 접근을 허용할지 직접 고르며, 한 번 켜면 모든 프롬프트에 기본 적용되지만 도구(Tools) 메뉴의 토글로 끌 수 있습니다. Google은 사용자 데이터로 모델을 학습하지 않는다고 밝혔습니다.

시점 정리는 다음과 같습니다. **2026년 4월** Nano Banana 기반 이미지 생성을 처음 발표했고, **2026년 6월 29일** 미국 적격 무료 사용자로 확대했습니다. 'Personal Intelligence' 기능 자체는 2026년 3월 미국에 폭넓게 출시된 뒤 인도와 일본으로 확대됐습니다. Gemini는 2026년 들어 월간 활성 사용자(MAU) 7억 5천만 명을 넘어섰습니다.
