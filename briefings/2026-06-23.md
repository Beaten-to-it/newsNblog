- **어디에 쓰나:** 오픈소스 유지보수 / 사내 보안 backlog / 취약점 관리
- **어떻게 적용하나:** 기존 scanner, CVE, bug bounty, 내부 ticket을 한 큐에 모은 뒤 AI가 중복 제거, 재현 가능성 확인, reachability 분석, 패치 초안, 테스트 생성을 맡게 합니다. 최종 판단은 maintainer나 보안 엔지니어가 하고, merge 전에는 재현 스크립트와 회귀 테스트를 필수 산출물로 요구합니다.
- **주의할 점:** AI가 취약점 리포트를 늘리는 순간 유지보수자 부담이 폭증합니다. false positive를 maintainer에게 넘기기 전에 전문가 검토와 priority 기준을 둬야 합니다.

### 2. [Gemini 앱을 Interactions API 기준으로 새로 설계하기](https://blog.google/innovation-and-ai/technology/developers-tools/interactions-api-general-availability/)

- **어디에 쓰나:** 에이전트 앱 / 리서치 자동화 / 장시간 백그라운드 작업 / 멀티툴 워크플로
- **어떻게 적용하나:** 신규 Gemini 기능은 단발 응답 API 대신 interaction 단위로 설계합니다. `background=True`로 오래 걸리는 작업을 분리하고, typed steps를 UI와 로그에 드러내며, Google Search·Maps·사내 function을 한 interaction에서 조합합니다.
- **주의할 점:** 상태를 서버에 저장하는 구조는 편하지만, 보존 기간·민감 데이터·재현성 정책이 필요합니다. 비용 최적화를 위해 Flex/Priority를 업무별로 나누는 기준도 먼저 정해야 합니다.

### 3. [Copilot/Claude/사내 에이전트를 IDE 안에서 조직 표준으로 배포하기](https://github.blog/changelog/2026-06-22-new-features-and-claude-as-agent-provider-preview-in-jetbrains-ides/)

- **어떻게 적용하나:** 조직 관리자가 반복 업무별 custom agent를 만들고, IDE agent picker에서 바로 선택하게 합니다. 예를 들어 보안 리뷰 에이전트, API 마이그레이션 에이전트, 테스트 보강 에이전트를 나누고, Copilot CLI의 queue/steer 기능으로 긴 작업 중간에 방향을 바꿀 수 있게 합니다.

## 8. 오늘의 관전 포인트

1. **보안 에이전트가 "리포트 생성기"를 넘어 실제 patch landing까지 책임질 수 있는가** — Daybreak의 성패는 벤치마크 점수보다 maintainer가 감당 가능한 품질의 패치와 테스트를 얼마나 내는지에 달렸습니다.
