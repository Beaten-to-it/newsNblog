### 1. [Claude Code 요청 마킹 논쟁](https://news.ycombinator.com/item?id=48734373)

- **무슨 이야기인가:** Claude Code가 proxy와 환경 조건을 시스템 컨텍스트에 간접적으로 표시한다는 분석이 공유됐습니다. 토론은 "anti-abuse fingerprinting은 필요하다"와 "개발자 도구가 은밀히 컨텍스트를 변조하면 신뢰가 깨진다"로 갈렸습니다.
- **주요 관점:** 로컬 에이전트는 일반 SaaS 클라이언트보다 높은 권한을 받기 때문에 telemetry와 policy enforcement의 투명성이 더 중요합니다. 특히 proxy, gateway, 사내 모델 라우터를 쓰는 팀은 클라이언트 업데이트가 요청 컨텍스트를 어떻게 바꾸는지 확인해야 합니다.
- **볼 만한 이유:** agent CLI가 local runtime이자 vendor policy enforcement point가 되는 순간, 보안 검토 범위는 모델 API 문서만으로 끝나지 않습니다.
- **관련 출처:** [원문 분석](https://thereallo.dev/blog/claude-code-prompt-steganography)

### 2. [Claude Sonnet 5의 cost-performance 논쟁](https://news.ycombinator.com/item?id=48736605)

- **무슨 이야기인가:** Sonnet 5 발표 뒤 HN에서는 medium effort 이상의 cost-performance, Opus 4.8과의 경계, GLM 계열 등 저가 모델과의 비교가 이어졌습니다. 단순 성능보다 effort level별 실제 비용과 latency가 모델 선택 기준으로 올라왔습니다.
- **주요 관점:** "모델이 좋아졌다"보다 "어느 effort에서 어떤 task를 끝내는가"가 중요합니다. Sonnet 5가 기본 모델이 되더라도 고난도 작업에서는 higher effort보다 모델 전환이 더 나은 경우가 생길 수 있습니다.
