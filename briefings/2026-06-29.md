- **어떻게 적용하나:** 작업을 난이도와 민감도 기준으로 분류하고, 기본 모델·저가 모델·고성능 모델을 나눕니다. 모델 호출 앞단에 라우터를 두고, context 길이, 캐시 hit rate, 재시도율, 사용자별 비용, 업무 산출물을 함께 집계합니다.
- **주의할 점:** 자동 라우팅은 품질 책임을 없애지 않습니다. 민감 데이터는 저가 외부 모델로 보내지 않도록 data class를 먼저 정의하고, 모델 전환 시 도구 호출 방식과 context window 차이를 테스트해야 합니다.

### 2. [AI 에이전트용 workspace sandbox를 기본값으로 운영하기](https://github.com/openai/codex/issues/2847)

- **어디에 쓰나:** 개발팀 로컬 환경 / CI 기반 코드 수정 에이전트 / 보안 리뷰 / 외주 코드 검토
- **어떻게 적용하나:** 에이전트가 읽을 수 있는 디렉터리를 작업 repo와 필요한 dependency cache로 제한합니다. `.env`, credential store, 개인 홈 디렉터리, SSH key, cloud config는 workspace 밖에 두고, 필요 시 read-only mount나 ephemeral VM으로 작업을 분리합니다.
- **주의할 점:** `.agentignore`류 파일은 유용한 힌트일 수 있지만 보안 장벽은 아닙니다. shell 명령 출력으로 내용이 우회 노출될 수 있으므로 실제 차단은 OS 권한, container, VM, network egress policy에서 해야 합니다.

### 3. [숙련자 지식을 AI 품질 검증 루프에 넣기](https://techcrunch.com/2026/06/28/ford-rehires-gray-beard-engineers-after-ai-falls-short/)
