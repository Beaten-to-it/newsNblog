## 4. AI UseCase

> 오늘 뉴스에서 바로 참고할 만한 실제 적용 아이디어만 짚습니다.

### 1. [모델 등급을 업무 위험도와 비용에 맞춰 라우팅하기](https://openai.com/index/previewing-gpt-5-6-sol/)

- **어디에 쓰나:** 사내 AI 플랫폼 / 코딩 에이전트 / 리서치 자동화 / 전략 문서 작성
- **어떻게 적용하나:** 업무를 low-risk draft, 반복 코드 수정, 보안·법무·재무 검토, 장시간 agent task로 나누고 각 단계에 기본 모델, 강한 모델, 승인 필요 모델을 배정합니다. 강한 모델 호출은 자동 로그, 비용 cap, 결과 검토 큐를 함께 묶습니다.
- **주의할 점:** 최고 모델 접근권을 사람 단위로만 주면 비용과 보안이 흐려집니다. 사용자 권한보다 task type, data sensitivity, external action 여부를 기준으로 라우팅해야 합니다.

### 2. [에이전트용 가입·권한 위임 플로우 설계하기](https://workos.com/auth-md)

- **어디에 쓰나:** SaaS onboarding / B2B API / marketplace / 업무 자동화 에이전트
- **어떻게 적용하나:** 사람이 계정을 만든 뒤 에이전트에게 제한된 scope, 만료 시간, 감사 로그가 붙은 권한을 위임하도록 설계합니다. 서비스 문서에는 에이전트가 따라야 할 signup endpoint, permission request, user confirmation 절차를 명확히 둡니다.
- **주의할 점:** 에이전트가 사람처럼 가입 폼을 우회해 계정을 만들게 두면 약관, 결제, 개인정보 동의, bot abuse 문제가 한꺼번에 생깁니다. 에이전트 전용 권한 경계와 revoke 흐름이 필요합니다.

### 3. [AI code review를 위험 기반으로 운영하기](https://github.blog/changelog/2026-06-25-copilot-code-review-analysis-depth-and-efficiency-updates/)

- **어디에 쓰나:** 개발팀 PR review / 보안 리뷰 / 데이터 마이그레이션 / release gate
- **어떻게 적용하나:** PR 라벨이나 파일 경로에 따라 리뷰 깊이를 다르게 둡니다. 결제, 인증, 권한, 암호화, migration 파일은 깊은 분석을 켜고, 문서나 단순 UI copy는 빠른 리뷰로 처리합니다.
- **주의할 점:** AI 리뷰를 필수 게이트로 만들면 false positive가 병목이 될 수 있습니다. 사람 리뷰어가 override할 수 있는 규칙과, AI가 놓친 결함을 다시 학습하는 postmortem 루프가 필요합니다.
