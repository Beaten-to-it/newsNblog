# Publishing Plan — Email + Blog

Day 1 브리핑을 예쁘게 메일로 발송하고 블로그에 올리기 위한 계획입니다.

## 현재 상태

- 브리핑 원문: `briefings/2026-06-05.md`
- 메일 발송 도구 상태:
  - `himalaya`: 설치되어 있지 않음
  - Google Workspace OAuth: 인증되어 있지 않음
- 블로그 플랫폼: 아직 미정
- 실제 메일 발송 / 블로그 게시: 사용자 확인 전에는 실행하지 않음

## 원칙

v0에서는 다시 복잡해지지 않도록 아래까지만 한다.

1. Markdown 브리핑을 예쁜 HTML 메일로 변환한다.
2. 같은 내용을 블로그용 Markdown/HTML로 정리한다.
3. 테스트/미리보기 파일을 만든다.
4. 사용자가 확인한 뒤에만 실제 메일 발송/블로그 게시를 한다.

## 메일 디자인 방향

- 제목: `오늘의 AI Morning Radar — YYYY.MM.DD`
- 폭: 640px 내외
- 모바일에서 읽기 쉬운 단일 컬럼
- 섹션별 카드형 구분
- 링크 색상 명확히 표시
- 불필요한 발행 이력/중복 체크/피드백 버튼 없음
- 텍스트 메일 fallback도 같이 생성

## 블로그 디자인 방향

- 제목 링크 유지
- GeekNews 개별 링크 유지
- 섹션 목차 추가 가능
- 블로그용 SEO 제목 별도 생성
- 블로그 공개용이므로 너무 개인 피드백/내부 운영 내용은 제거

## 필요한 사용자 입력

### 1. 메일 발송 방식

하나 선택:

- Gmail/Google Workspace OAuth
- SMTP/Himalaya
- 지금은 HTML 파일만 만들고 수동 발송

### 2. 받을 이메일 주소

예:

```text
받을 주소: example@gmail.com
```

### 3. 블로그 플랫폼

하나 선택:

- GitHub Pages / 정적 블로그
- WordPress
- Ghost
- Notion
- 티스토리
- 네이버 블로그
- 아직 미정 — 우선 블로그용 Markdown만 생성

## 추천 v0 진행

가장 덜 복잡한 순서:

1. `dist/email/2026-06-05.html` 생성
2. `dist/blog/2026-06-05.md` 생성
3. 사용자가 눈으로 확인
4. 메일은 우선 수동 발송 또는 Gmail 인증 후 발송
5. 블로그는 플랫폼이 정해질 때까지 Markdown 아카이브만 유지

## Kanban 게이트

메일 발송과 블로그 게시는 외부 side effect가 있으므로 아래 정보가 없으면 blocked 상태로 둔다.

- 메일 수신자
- 메일 발송 계정/도구
- 블로그 플랫폼
- 게시 공개 여부
- 최종 발송/게시 승인
