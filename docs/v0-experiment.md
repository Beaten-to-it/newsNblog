# AI Morning Radar v0 — 7일 실험

## 목적

이 실험의 목적은 AI 뉴스 서비스 MVP를 만드는 것이 아닙니다.

목적은 단 하나입니다.

> 매일 아침 받아보는 AI 브리핑이 실제로 유용한지, 어떤 포맷이면 계속 읽게 되는지 확인한다.

## 검증 질문

1. 나는 7일 중 5일 이상 이 브리핑을 실제로 읽는가?
2. 7일 중 3일 이상 “오늘 내용은 유용했다”고 느끼는가?
3. X/Threads 중심의 빠른 트렌드 섹션이 실제로 유용한가?
4. GeekNews 같은 국내 개발자 큐레이션 소스가 브리핑 품질을 올리는가?
5. 한 번 낸 뉴스를 중복 제외했을 때 정보 밀도가 좋아지는가?
6. 어떤 섹션을 유지하고, 어떤 섹션을 줄이거나 제거해야 하는가?
7. 블로그에 올릴 만큼 아카이브 가치가 있는가?

## v0에 포함하는 것

- 한국어 AI 뉴스 브리핑 Markdown 작성
- 공식 발표/주요 미디어/GeekNews/개발자 커뮤니티/웹상 화제 기반 후보 수집
- X/Threads 이슈를 처음부터 주요 소스로 포함
- X/Threads는 최신 트렌드 민감도가 높으므로 브리핑 내 비중을 높게 배정
- 이미 발행한 뉴스는 `data/published_items.csv` 기준으로 중복 제외
- 각 뉴스에 `왜 중요한가` 포함
- 신뢰도 표시: 높음 / 중간 / 낮음
- 피드백은 본문에 질문/버튼으로 넣지 않고, 사용자가 대화에서 자연스럽게 주는 내용만 `data/feedback.csv`에 수동 기록한다.
- 피드백을 `data/feedback.csv`에 기록

## v0에서 제외하는 것

- 블로그 자동 게시
- 이메일 자동 발송
- 좋아요 버튼/폼/자동 피드백 수집 구현
- 추천 알고리즘
- 데이터베이스
- 관리자 페이지
- 사용자 계정

> 단, X/Threads는 v0부터 내용 소스로 포함한다. API/자동화 방식은 상황에 따라 웹 검색, 공개 페이지 확인, 수동 링크 입력, 나중의 API 연동 후보로 나눈다.

## 수집 비중 가이드

v0 브리핑 후보 수집은 대략 아래 비중을 기준으로 한다.

```text
X / Threads 실시간 트렌드: 40%
공식 블로그 / 제품 발표: 25%
GeekNews / Hacker News / GitHub: 20%
주요 미디어 / 정책 / 비즈니스: 15%
```

단, 최종 Top 5에는 신뢰도와 중요도를 우선한다. X/Threads발 루머는 신뢰도 낮음으로 표시하고, 공식 확인이 없으면 Top 5가 아니라 트렌드 섹션에 둔다.

## 중복 방지 규칙

매일 브리핑 작성 전 `data/published_items.csv`를 확인한다.

중복으로 간주하는 경우:

- 같은 URL을 이미 발행함
- 같은 공식 발표를 다른 매체가 재보도한 기사
- 제목만 다르고 핵심 내용이 같은 X/Threads 논의
- 같은 모델/제품/회사 발표를 전날 이미 다룸

중복이지만 후속 보도가 의미 있는 경우에는 새 항목으로 낼 수 있다. 단, 제목에 `후속` 또는 `업데이트`를 표시하고 기존 항목과 달라진 점을 명확히 쓴다.

예:

```text
중복 제외: OpenAI 모델 발표를 다룬 TechCrunch 재보도
발행 가능: OpenAI 발표 후 가격/벤치마크/개발자 반응이 새로 확인된 후속 분석
```

## 일일 산출물

파일명:

```text
briefings/YYYY-MM-DD.md
```

각 브리핑은 다음 섹션을 포함합니다.

1. 한 줄 요약
2. 빠른 트렌드 레이더 — X/Threads/GeekNews
3. 오늘 꼭 봐야 할 AI 뉴스 Top 5
4. X/Threads 주요 논의 자세히 보기
5. 개발자/제품 체크포인트
6. 비즈니스/정책 체크포인트
7. 오늘 제외한 중복/저신뢰 이슈
8. 오늘의 관전 포인트
9. 사용자 피드백은 대화에서 들어온 경우에만 `data/feedback.csv`에 수동 기록
10. 출처 링크

## 피드백 방식

v0에서는 브리핑 본문에 피드백 질문, 좋아요 버튼, 폼 링크를 넣지 않습니다.
사용자가 대화에서 자연스럽게 피드백을 주면 아래 정보를 `data/feedback.csv`에 수동 기록합니다.

- 읽었는가? yes/no
- 유용도: 1~5, 제공된 경우만
- 좋았던 항목, 제공된 경우만
- 별로였던 항목, 제공된 경우만
- 내일 더 보고 싶은 주제
- X/Threads 비중이 적절했는가?
- 중복이 느껴진 항목이 있었는가?
- 메모

## 성공 기준

v0는 아래 조건을 만족하면 성공입니다.

- 7일간 7개 브리핑 생성
- 최소 5회 이상 실제로 읽음
- 최소 3회 이상 유용도 4점 이상
- 중복 뉴스가 체감상 크게 줄었음
- X/Threads/GeekNews 포함이 브리핑 가치를 높였는지 판단 가능
- v1에서 자동화할 항목을 1~3개로 좁힘

## 실패 기준

아래에 해당하면 v0는 실패가 아니라 “중단할 근거를 얻은 것”입니다.

- 7일 중 3일 이하만 읽음
- 대부분 내용이 길거나 피로함
- X/Threads 비중이 높아져서 루머/노이즈가 과해짐
- 중복 체크 비용이 가치보다 큼
- 블로그 아카이브로 남길 필요를 못 느낌

## v1 후보

v0 결과가 좋을 때만 아래 중 최대 3개를 선택합니다.

- 이메일 자동 발송
- 블로그 Markdown → 게시 자동화
- X/Threads 수집 방식 고도화
- GeekNews/RSS 자동 수집
- 소스 목록 고정 및 자동 수집
- 피드백 CSV 기반 주제 가중치 반영
- 발행 이력 기반 중복 체크 자동화
- 피드백 수집/개인화는 v0 이후에만 재검토
